## 1. 总体建议（结论先行）

**最终结论：选 F + C，按顺序执行。**

1. **F（先收尾 mini-mall AI 库存助手）**：用 1–2 周把 mini-mall Phase 3 收尾，**重点是 admin 前端 AI 页面（当前为零）**。后端已写 3071 行却没有任何可演示界面，这是简历上最致命的状态——"做了但展示不出来"等于没做。收尾期间**严禁**给 mini-mall 增加任何新范围。
2. **C（独立开发"项目文档 RAG 助手"）**：mini-mall Phase 3 验收通过后启动，独立仓库、独立技术栈（Spring Boot 单体 + Spring AI + PostgreSQL/pgvector），4–5 周完成 MVP。**用 mini-mall 自己的 26+ 份真实文档（README、architecture.md、各 Phase 契约、dev-log）作为种子语料和演示数据**——这同时解决了数据来源问题和简历叙事问题。

不选 A/B/D/E 作为主形态的原因见第 4 节；与 mini-mall 的关系采用"方案 3：先独立，后场景化接入"，见第 6 节。

### 三个判断

| 问题 | 结论 | 理由 |
|---|---|---|
| 值得独立开发吗 | **值得** | RAG 是 AI 应用岗/Java 后端 AI 方向面试出现频率最高的工程问题；mini-mall 的 AI 库存助手是"结构化数据 + Prompt + 输出校验"路线，完全不覆盖 embedding / 向量检索 / 文档解析 / 引用溯源，两者互补不重复 |
| 应该整合进 mini-mall 吗 | **不应该** | mini-mall 已 10 模块 + 12 容器 + MySQL；RAG 需要 PostgreSQL/pgvector，整合意味着双数据库、更多容器、更脆的答辩演示，且 Phase 3 还没收尾。两个边界清晰的项目 > 一个臃肿项目 |
| 应该暂缓吗 | **暂缓 1–2 周，不是无限期** | 唯一暂缓理由是 Phase 3 前端为零。收尾有明确边界（TaskMaster phase3 剩余 7 任务），不会变成无限拖延 |

## 2. 是否会和 AI 库存助手重复？（关键质疑，先自答）

不重复，且面试时这是一个**主动讲的对比点**：

| 维度 | mini-mall AI 库存助手 | RAG 文档助手 |
|---|---|---|
| 数据形态 | 结构化（库存、销量、订单证据） | 非结构化（PDF/Markdown 文档） |
| 核心技术 | 证据门面 + 版本化 Prompt + JSON 输出校验 + 建议审批流 | 文档解析、切块、embedding、向量检索、引用溯源 |
| 防幻觉手段 | 结构化证据注入 + Schema 校验 | 检索阈值 + 引用标号强制 + 无答案路径 |
| 写权限 | 生成建议草稿，人工确认后才动库存 | 纯只读问答 |

面试一句话：「我做过两种 AI 落地形态：一种是把结构化业务数据喂给模型做决策建议并用 Schema 校验输出；另一种是对非结构化文档做 RAG 检索问答并强制引用来源。」这句话能撑起 10 分钟追问。

## 3. 核心目标分析（按你列的 9 个角度）

1. **简历价值：高。** Java 后端 + RAG 完整链路是 2025–2026 校招稀缺组合；多数学生只有"调 API 的聊天机器人"。
2. **后端能力展示：高。** 异步 ingestion 状态机、失败重试、pgvector 索引与过滤、流式响应、调用日志——全是后端工程问题，不是 AI 玄学。
3. **AI 应用能力展示：高。** 但前提是做出引用溯源、无答案处理、评测集这三件事，否则就是套壳。
4. **与 mini-mall 互补：是。** 一个讲分布式工程化，一个讲 AI 应用工程化。
5. **与 AI 库存助手重复：否。**（见第 2 节）
6. **适合实训/独立展示：适合。** 单体 + Compose 三条命令冷启动，演示比 12 容器的 mini-mall 稳得多。
7. **2–6 周 MVP：可行。** 按规划 03 的 Phase 0–5+7 约 3.5–4.5 周，留有缓冲。
8. **会不会变成套壳 ChatBot：会，如果你不做这五件事** —— 引用来源、无答案路径、ingestion 可观测、检索调试页、评测集。本规划把这五件全部放进 MVP 必做。
9. **经得起面试追问：能。** 因为关键决策（pgvector vs Milvus、自写检索 vs 框架黑盒、不做 rerank 的理由）都是有据可讲的取舍，见规划 04。

## 4. 候选方向比较与排序

### 排序：C > D（作为 C 的语料场景，不作为独立形态） > E（作为 C 的 Phase 6 增量） > A > B

### C. 项目文档 RAG 助手 ★★★★★（选定）

- **亮点**：语料是你自己的真实工程文档（mini-mall 26+ 份 docs + PRD + 契约 + dev-log），问答有真实答案可验证；天然形成"我给自己的微服务项目造了一个文档问答系统"的叙事闭环。
- **难度**：中。无爬虫、无 OCR、无多租户。
- **简历价值**：高，且与 mini-mall 形成"二项目组合拳"。
- **套壳风险**：低——只要做引用与评测。
- **MVP 范围**：单用户登录、多知识库、PDF/MD/TXT 上传、异步 ingestion、向量检索、带引用问答、无答案处理、调用日志后台。
- **技术栈**：Spring Boot 3 + Spring AI + pgvector + Vue 3（详见规划 02）。
- **最大风险**：PDF 解析质量参差（规避：演示语料以 Markdown 为主，PDF 为辅）。

### D. mini-mall 内部运营知识库 ★★★（场景化吸收，不独立做）

- **亮点**：业务故事好讲。
- **致命问题**：作为 mini-mall 子服务做会引入双数据库 + 容器膨胀 + Phase 3 未完的三重风险；mini-mall 2026-06-10 架构体检已显示项目处于"中期半成品"状态，再加模块是失控路线。
- **处置**：D 不独立成项目，而是 C 的一个演示知识库——在 C 里建一个名为"mini-mall 运营知识库"的 KB，导入 mini-mall 文档。简历上表述为应用场景，不表述为架构耦合。

### E. Agent + RAG 工作流助手 ★★★（作为 Phase 6 可选增量）

- **亮点**：差异化高。
- **风险**：以 Agent 为主目标几乎必然过度设计；学生项目里"会调工具的 Agent"往往演示 3 分钟、调试 3 周。
- **处置**：MVP 不做。Phase 6 限定为"2–3 个只读工具 + 单一用例（根据项目文档生成开发任务清单）+ tool_call 日志 + 只生成不执行"。

### A. 校园智能助手 ★★

- **致命问题**：数据来源。校规、考试安排、办事流程要么爬（合规与时效风险），要么手工编（演示数据是假的，答辩一问"数据哪来的"就塌）。受众故事好讲但工程上站不住。
- **处置**：不选。若学校实训题目强制要求"校园"主题，可以把 C 换皮：同一套系统，知识库换成"教务处公开 PDF + 课程讲义"，架构与代码 100% 复用。

### B. 个人学习知识库 ★

- **致命问题**：同质化最严重的方向，Notion AI / ima / 各种开源克隆已泛滥；面试官见过几十个，没有提问欲望。
- **处置**：不选。

## 5. 技术路线比较与推荐

### 三条路线对比

| 维度 | 路线 1：Python + FastAPI + LangChain | 路线 2：Spring Boot + Spring AI + pgvector | 路线 3：mini-mall 扩展模块 |
|---|---|---|---|
| 学习成本 | 高（新语言生态、依赖地狱） | 低（你已熟 Spring 全家桶） | 低但有隐性成本（双库、契约联动） |
| 开发速度 | 框架快，但调试慢 | 中等偏快 | 最慢（受 mini-mall 现状牵制） |
| 工程展示价值 | 中（LangChain 胶水代码是面试减分常见项） | 高（Java+AI 稀缺组合，且检索层自己写） | 中（淹没在已有 10 模块里） |
| 与现有栈一致性 | 断裂 | 高度一致（JWT/分层/测试习惯全复用） | 完全一致但拖累宿主 |
| 简历适配 | 适合纯 AI 岗 | 适合 Java 后端 + AI 应用岗（你的画像） | 弱化为"又加了个功能" |
| 后续维护 | 你不熟 Python 工程化，维护差 | 好 | 差（耦合） |
| Codex/Opus 执行友好度 | 高 | 高（Spring 模板代码模型非常擅长） | 低（需理解 mini-mall 全量上下文） |
| 失控风险 | 中（LangChain 抽象漂移） | 低 | 高 |

### 最终推荐：路线 2，且有三条强制约束

1. **单体单模块**。一个 Spring Boot 应用 + 一个 Vue 前端 + 一个 PostgreSQL。不拆微服务、不上 Redis、不上 RabbitMQ、不上 MinIO（文件落本地卷，留 `StorageService` 接口以便后换）。微服务工程化你已经用 mini-mall 证明过了，这里再拆是重复劳动 + 失控风险。
2. **Spring AI 只用两件事：`ChatClient`（含流式）和 `EmbeddingModel`。** 向量存储与检索**不用** Spring AI 的 `PgVectorStore`（它强制自带表结构和 jsonb metadata，会让你失去 citation 所需的强类型 chunk 元数据），改为自己建 `document_chunks` 表 + JDBC 写检索 SQL。这是面试上的关键差异点：「检索层我没有用框架黑盒，SQL、索引、过滤都是自己写的」。
3. **Provider 全部走 OpenAI 兼容协议 + Mock 兜底**（复用你在 mini-mall 已验证的模式）：
   - LLM：DeepSeek-chat（你已有 key、已有适配经验，OpenAI 兼容）。
   - Embedding：DeepSeek 不提供 embedding，推荐 SiliconFlow 的 `BAAI/bge-m3`（OpenAI 兼容端点、1024 维、有免费额度）或阿里 DashScope `text-embedding-v4`。二选一即可，接口抽象成 `EmbeddingProvider`。
   - Mock Provider 为默认配置：无 key 也能跑通全链路（mock embedding 用内容哈希生成确定性向量），保证测试与离线演示。

### 推荐技术栈清单（最终版）

| 层 | 选型 | 不选什么及理由 |
|---|---|---|
| 后端 | Java 17 + Spring Boot 3.x 单体 | 不用 Spring Cloud：单体无此需求 |
| AI 接入 | Spring AI（ChatClient + EmbeddingModel，OpenAI 兼容） | 不用 LangChain4j：Spring AI 是 Spring 官方项目，简历关键词更硬 |
| 数据库 | PostgreSQL 16 + pgvector 扩展（关系数据与向量同库） | 不用 Milvus/ES：<10 万 chunk 规模下 HNSW 足够，少两个容器，事务一致 |
| 文档解析 | Apache Tika（PDF/TXT/MD/DOCX 统一入口） | 不做 OCR：扫描件明确拒收 |
| 迁移 | Flyway | 与 mini-mall 手工迁移区分，展示另一种实践 |
| 文件存储 | 本地磁盘卷 + StorageService 接口 | MinIO 暂缓：少一个容器 |
| 前端 | Vue 3 + Vite + Pinia + Element Plus | 与你 admin-frontend 技术一致，零学习成本 |
| 流式 | SSE（Spring AI 流式 ChatClient → EventSource） | 展示版做，MVP 先同步 JSON |
| 部署 | Docker Compose（postgres + backend + frontend 共 3 容器） | 坚决不上 K8s |
| 可观测 | model_call_logs / ingestion_jobs 表 + 后台页面 | 不上 Prometheus/Grafana：mini-mall 已展示过 |

## 6. 与 mini-mall 的关系（三方案分析）

### 方案 1：完全独立
- 优点：边界清晰、互不拖累、答辩各自独立、简历两段叙事。
- 缺点：演示语料需准备；少一层业务故事。
- 适合：默认底线方案。

### 方案 2：作为 mini-mall 的 knowledge-service
- 优点：单一大项目叙事；复用网关、JWT、审计。
- 缺点：MySQL 之外引入 PostgreSQL 双库；compose 从 12 容器继续膨胀；Phase 3 未完先加新服务是典型失控；答辩演示链路更长更脆；RAG 的亮点淹没在电商模块里。
- 适合：仅当 mini-mall 已全部收尾、且实训强制要求单项目时。**现在不适合。**

### 方案 3：先独立开发，后场景化接入（★ 推荐）
- 做法：RAG 项目独立仓库独立部署；建一个"mini-mall 运营知识库"KB 导入 mini-mall 的 docs/、PRD、契约文档；演示时现场问"订单超时未支付怎么处理""库存锁定的幂等怎么做的"并展示引用到具体文档段落。可选的最浅集成：admin-frontend 加一个外链菜单项指向 RAG 系统（一行路由，不合并任何代码）。
- 优点：拿到方案 1 的全部边界收益 + 方案 2 的叙事收益；零架构耦合；两个项目在答辩里互相佐证（"我的文档工程化好到可以直接喂给 RAG"）。
- 缺点：集成深度浅——但对学生项目这是优点不是缺点。
- **结论：采用方案 3。简历表述为应用场景（"已用于索引本人微服务项目的 30+ 份工程文档"），不表述为系统集成。**
