## 1. 简历描述（四个版本，按实际完成形态选用）

写作纪律：不写"商业项目"、不写"训练大模型"、不写虚构用户量；每一条都要能现场演示或翻代码佐证。

### 版本 1：独立 RAG 知识库项目（MVP 完成即可用）

> **项目文档智能问答系统（个人项目）** — Java 17 / Spring Boot 3 / Spring AI / PostgreSQL+pgvector / Vue 3 / Docker Compose
> - 设计并实现完整 RAG 链路：文档上传 → Tika 解析 → 标题感知切块（600–900 字、120 字重叠）→ 批量 Embedding → pgvector HNSW 向量检索 → 引用溯源问答；已用于索引本人微服务项目的 30+ 份工程文档。
> - 实现基于数据库任务表的异步 ingestion 状态机（解析/切块/向量化三阶段、失败重试、错误可视化），保证文档处理幂等可重跑。
> - 通过相似度阈值短路、强约束 Prompt 模板、引用编号校验三层机制控制幻觉；知识库无答案时显式拒答；25 题人工评测集验证有据回答率 ≥80%。
> - 向量检索层不依赖框架黑盒，自行设计 chunk 元数据表与检索 SQL，实现多知识库 kb_id 隔离与属主权限过滤。
> - 全链路可观测：模型调用日志（token/延迟/状态）、ingestion 任务日志、检索调试工具；三容器 Docker Compose 一键部署。

## 2. 面试追问与回答方向

| 问题 | 回答骨架 |
|---|---|
| RAG 和普通 ChatBot 区别？ | ChatBot 答案来自参数记忆，无法溯源、无法及时更新；RAG 把"知识"外置到向量库，回答前先检索，答案受上下文约束且每句可定位到文档段落。我的系统里区别是可演示的：同一问题，库内有文档→带引用回答，删掉文档→显式拒答。 |
| chunk 策略？ | 两套：Markdown 按标题树切并保留 heading_path（引用展示直接受益）；PDF/TXT 段落滑窗 600–900 字、120 字重叠。参数不是拍的——用检索调试页对 10 个真实 query 人工核过 top-8 命中。能继续讲：过碎丢上下文、过大稀释相似度、重叠解决跨块语义断裂。 |
| 为什么 pgvector 不用 Milvus？ | 规模 <10 万 chunk，HNSW 在 PG 内性能足够；向量与业务元数据同库带来事务一致（删文档级联删向量，无分布式一致性问题）；少两个容器。同时我知道边界：HNSW+WHERE 是后过滤，库大后召回会衰减，到百万级才值得上专用向量库，且我把检索收口在一个接口后面，可替换。 |
| 如何保证回答有依据？ | Prompt 强制每个论断标 `[n]`；后端校验编号合法性并落 citations 表（含原文快照）；无引用的回答标 UNGROUNDED 在 UI 上明示。 |
| 如何降低幻觉？ | 五层：检索阈值短路（最关键，不给模型编的机会）→ 低 temperature → 模板三规则 → 引用校验 → 评测集验收。注意第一层在调模型之前，还省成本。 |
| 知识库没答案怎么办？ | top1 相似度低于标定阈值直接返回固定拒答话术（不调 LLM）；阈值之上但资料不足时模板规则要求模型拒答；两种都记 NO_ANSWER 状态，评测集里有 5 道库外题专测这条路径。 |
| 权限隔离怎么做？ | 两级：service 层先校验 KB 属主，检索 SQL 强制 `WHERE kb_id=?`（kb_id 冗余到 chunk 表免 JOIN）；有"KB-A 检索不出 KB-B"的集成测试。没做 RBAC——单角色场景做权限矩阵是过度设计，这个判断本身就是答案的一部分。 |
| 文档更新后向量如何同步？ | MVP 明确不做原位更新，只做删除+重传：chunk 对 document 外键 CASCADE，向量同行即删，不存在脏向量；citations 留快照所以历史问答不坏。版本化更新是规划好的增强项，不是没想到。 |
| 为什么不把全文塞给模型？ | 成本随库线性增长且有上下文上限；长上下文中部信息衰减；无法定位来源。RAG 每次只进 top-6 块，成本 O(k)、来源可溯、文档更新即时生效。 |
| Agent 部分做了什么？ | （没做 Phase 6 时）"我刻意没做——问答闭环没有评测基础前上 Agent 是演示风险"，这个取舍讲出来是加分。（做了时）按规划 02 §8 讲：3 只读工具、5 步上限、只生成不执行、全量 tool 日志。 |
| 和 mini-mall 什么关系？ | 架构完全独立、零代码耦合；mini-mall 的 30+ 工程文档是它的种子语料和演示场景。两个项目覆盖两种 AI 落地形态：结构化数据决策建议（库存助手）vs 非结构化检索问答（RAG）。 |
| 最大技术难点？ | 文档解析与切块质量——它决定检索上限而又没有银弹。我的工程答案：失败显式化（坏 PDF 直接 FAILED 给原因）、参数用真实 query 标定、检索调试页让质量问题可观察可归因。 |

## 3. 最终演示脚本（5 分钟，按此排练）

0. 开场一句话：「给我自己的微服务项目造的文档问答系统，重点看三件事：引用溯源、无答案处理、全链路可观测。」
1. `docker compose up -d` 已起（或现场起，掐表 <1 分钟）；登录。
2. 展示"mini-mall 项目文档"知识库：8–10 份真实文档，全部 READY。
3. 现场上传一份新 MD → 状态徽标 UPLOADED→…→READY（~10 秒）。
4. 提问 1（库内）：「订单超时未支付是怎么处理的？」→ 带 `[1][2]` 引用的回答 → 点开引用卡片对照原文。
5. 提问 2（库外）：「学校食堂几点开门？」→ 显式拒答，强调"没有编"。
6. 后台：模型调用日志（指出刚才两次调用的 token 与延迟；第 5 步如果走了阈值短路，指出它根本没调模型）。
7. 失败演示：上传损坏 PDF → ingestion 页 FAILED + 可读原因 → 点"重新解析"演示幂等重跑（用修复后的文件）。
8. 检索调试页：输入 query 给面试官看 top-8 原始 chunk 与相似度分数——「这就是模型回答前看到的东西」。
9. 收尾一句话：架构图 + 「评测集 25 题，有据回答率 X%」。

备份：全程录屏一份，防现场网络/供应商抖动；Mock 模式作为最后兜底。
